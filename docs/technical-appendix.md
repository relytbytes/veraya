## 1. Demand forecasting (`lib/forecast.ts`, `lib/forecast-day.ts`)

A single-source day forecast (`computeDayForecast`) consumed identically by the dashboard projection and the Forecast card, so the two can never diverge.

**Point forecast.** Same-day-of-week historical samples are grouped (`groupSameDowSamples`) and combined as a **recency-weighted seasonal mean** (exponential decay on sample age) plus a **damped linear trend** — the damping prevents a short hot/cold streak from extrapolating unboundedly. Exogenous signals are applied multiplicatively:

```
projected = seasonalMean(recencyWeighted) × (1 + dampedTrend)
            × holidayMultiplier × weatherMultiplier
            + reservationLift + eventLift
```

**Reservations & events.** Confirmed covers and event headcount are converted to an incremental revenue lift rather than blended into the mean, so a heavily-booked night projects above baseline without polluting the seasonal history.

**Per-daypart decomposition.** `forecastDayparts` distributes the day total across services using each daypart's learned revenue share, so prep and labor can be timed, not just sized.

**Prep quantities — newsvendor / critical-fractile.** Prep targets are a **quantile** of the demand distribution, not the mean. Given an under-stock vs. over-stock cost ratio, the optimal service level is the critical fractile `Cu / (Cu + Co)`; we invert the normal CDF (`invNormCdf`, Acklam/Beasley-Springer-Moro-style rational approximation) at that fractile, scaled by forecast variance, to set prep above the mean by the economically-correct buffer.

**Confidence.** Variance-aware: confidence is a function of sample count and dispersion of the same-DOW samples, surfaced as honest bands rather than a single number.

**Tunable + self-correcting.** All coefficients live in `ForecastParams` (`lib/forecast-params.ts`). `backtestSales` replays history with a given parameter set and reports MAPE; the `forecast-tune` cron searches parameters against that backtest and persists the best set — a closed self-tuning loop. (Validated on synthetic patterned data: MAPE ~17% → ~6.5%.)

## 2. Health diagnosis & projection (`lib/vera-health.ts`)

`buildDiagnosis` scores five dimensions — profitability, demand, labor, cost, service — each `[0,100]`, then blends to an overall score using **learned, confidence-weighted weights** (§4).

**Intraday projection waterfall.** The day's revenue projection blends the pre-service forecast with realized pace, weighted by how much of the day has elapsed on the **learned intraday curve** (§3):

```
observed   = clamp(expectedRevenueFractionByNow, 0, 1)
projected  = salesSoFar + baseFloor × (1 − observed)
```

where `baseFloor` is the forecast (or a reservation-lifted baseline). This deliberately avoids the naive `salesSoFar / elapsedFraction` pace estimator, which collapses toward zero during a quiet afternoon and produced false "on pace to lose money" alarms; the booked dinner still counts when the room is momentarily empty.

**P&L close.** Projected COGS / labor / opex / fixed are applied to get projected net and margin; break-even uses a contribution-margin denominator guarded against a misconfigured cost rate (`max(0.05, 1 − cogs − opex)`), so it can't divide by zero.

## 3. Learned baselines (`lib/vera-baselines.ts`)

From a trailing window (~84 days):
- **Per-DOW expected revenue** (`expectedRevenueForDow`) — each weekday's own mean.
- **Intraday cumulative curve** (`expectedFractionByNow`) — the empirically-learned share of a day's revenue normally booked by a given hour, used as the `observed` term in the projection waterfall.
- `zScore` normalizes today's figures against the learned mean/σ for comparative indicators.

Everything is venue-specific; there are no hard-coded "lunch = X%" rules.

## 4. Learned signal weighting (`lib/vera-weights.ts`)

`getLearnedWeights` correlates each health dimension's historical score against realized daily margin (from the snapshot store) using **Pearson correlation**, then converts correlations to dimension weights via a **Bayesian blend** toward a uniform prior:

```
weight_i ∝ priorStrength × uniform + observed × |corr_i|
```

The prior dominates until enough days accumulate (the "Vera is learning" phase, gated on `daysObserved ≥ minDays`), then the data takes over — so a new deployment degrades gracefully instead of overfitting early. This is how the system discovers, per venue, that (e.g.) the demand signal predicts margin more than the cost signal.

**Training data** is generated by `snapshotDay` (`lib/vera-snapshot.ts`), which at close records each dimension's realized score paired with that day's realized P&L. Void/comp dollars are computed from flagged order items (not the nullable audit amount), identical to the live route, so weights train on the same numbers the dashboard shows.

**Third loop:** `VeraFeedback` (staff helpful/dismiss) adjusts surfacing/ranking of indicators independent of the score math.

## 5. Exogenous signals

**Weather (`lib/weather.ts`).** `getWeatherDisplay` is the single fetch (Open-Meteo, no key); `getWeatherSignal` derives the forecast multiplier from it so the widget and forecast never disagree. `demandMultiplier` is a transparent two-sided comfort model — pleasant/dry days lift demand, rain (intensity-scaled) and temperature extremes suppress it — clamped to `[0.82, 1.08]`. Documented as a heuristic prior; the architecture logs weather alongside sales so true coefficients can be learned later.

**Calendar (`lib/calendar.ts`).** `usHoliday` classifies holidays as busy / slow / closed; `holidayMultiplier` maps to a prior. A "closed"-tagged holiday with reservations or events on the books is treated as open (strong reduction, not ~5%) — data overrides the static label.

## 6. Operational inference

**Service-stage inference (`lib/stage-inference.ts`).** `inferStageFromItems` maps the current order composition (by category) to a dining stage; `advanceStage` enforces monotonic progression (SEATED → APPS → ENTREES → DESSERT → CHECK → BUSSING) so the floor view advances itself.

**Guest recognition (`lib/customer-tags.ts`, `lib/guest-insights.ts`).** `applyAutoTags` assigns Regular/VIP from visit-count and recency thresholds; `getGuestInsights` aggregates per-guest favorites, per-person-average spend, dwell time, and visit cadence for the pre-shift report.

**Reservation engine (`lib/reservations.ts`).** Single authoritative source for overlap/capacity/conflict (`timesOverlap`, `freeTablesFor`, `tableHasConflict`, `bookReservation`), pacing/cover limits (`getPacingLimit`, `coversInWindow`), and availability (`getDayAvailability`). Card-hold settlement and no-show fees in `lib/reservation-fees.ts`; auto no-show sweep on cron.

**Vision / extraction.** Full-invoice extraction (vendor + every line + cost, fuzzy-matched to existing suppliers/ingredients, totals-validated) and specific-product label identification via GPT-4o vision at high detail.

## 7. Architecture notes relevant to diligence

- **Stack:** Next.js 16 (App Router, RSC) + Prisma 7 on libSQL/Turso (web, Vercel); Expo SDK 54 / React Native (iOS), OTA via EAS. Shared API; mobile is a thin client over the same routes.
- **Realtime:** Redis-backed pub/sub with scoped channels (floor / kitchen / data) bridged to SSE; React Query cache invalidation per scope; polling as graceful fallback.
- **Determinism:** money math rounds at defined points (weighted-average cost to 4 dp; totals to 2 dp); reservation logic is single-sourced; schema changes are migration-gated.
- **Learning data is first-party and cumulative** — the snapshot store is the proprietary training corpus and grows per venue; it is not derivable from a competitor's cold start.

## 8. Defensibility summary

The novel, hardest-to-replicate element is the **composed closed loop**: forecast → realized snapshot → correlation-learned weights → confidence-weighted diagnosis → staff-feedback surfacing → tuned forecast. Individual techniques (newsvendor prep, damped-trend seasonal, Pearson→Bayesian weighting) are known in isolation; their **integration into a per-venue self-improving restaurant operations model, plus the accumulated training corpus,** is the trade secret. Strongest provisional-patent candidates: the self-tuning forecast loop and the confidence-weighted multi-dimensional health-scoring + intraday projection-waterfall method.
